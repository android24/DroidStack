# 第 2.1 节：如何学习一个开源框架

第一章建立了 Android 开源框架的全景地图。接下来要解决一个更具体的问题：面对一个陌生框架时，应该如何学习？

很多同学学习框架时容易走向两个极端：

- 只看 Quick Start，能跑 Demo，但不知道框架为什么这样设计；
- 一上来就读源码，陷入大量类名、泛型、线程和平台细节，很快迷路。

更有效的方法是：先理解框架解决的问题，再识别核心抽象，最后沿着少数关键调用链进入源码。学习框架不是“从第一个文件读到最后一个文件”，而是建立从使用场景到内部机制的路径。

## 2.1.1 本节学习目标

学习完本节后，你应该能够：

- 用统一的问题清单快速理解一个开源框架；
- 从 README、官方文档、Sample 和测试中抓住框架主线；
- 识别框架的核心入口类、核心抽象和扩展点；
- 画出一个框架的最小调用链；
- 判断哪些源码值得优先阅读，哪些细节可以暂时跳过；
- 将框架学习笔记整理成可复用的工程知识。

## 2.1.2 学习框架先问四个问题

学习任何框架前，先不要急着写代码，也不要急着打开源码。先回答四个问题。

### 1. 它解决什么问题

这是最重要的问题。框架一定是为了解决某类重复出现的工程问题。

例如：

| 框架 | 解决的问题 |
| --- | --- |
| Coil / Glide | 网络图片加载、解码、缓存、生命周期绑定、列表复用 |
| OkHttp | HTTP 连接、请求调度、拦截器、连接池、缓存、超时和重试 |
| Retrofit | 将 HTTP API 描述为接口方法，并完成请求构建、调用适配和响应转换 |
| Room | 在 SQLite 之上提供类型安全的数据访问、编译期校验和迁移管理 |
| Hilt | 管理对象创建、依赖图、作用域和 Android 组件注入 |
| WorkManager | 调度需要可靠执行、可延迟、受约束条件控制的后台任务 |

如果你说不清框架解决什么问题，后面的 API 和源码都会变成零散记忆。

### 2. 它提供什么核心抽象

框架通常会把复杂问题封装成几个核心抽象。学习框架要先抓这些抽象，而不是背所有方法。

例如：

- OkHttp 的核心抽象是 `OkHttpClient`、`Request`、`Response`、`Call`、`Interceptor`；
- Retrofit 的核心抽象是接口方法、注解、`ServiceMethod`、`CallAdapter`、`Converter`；
- Room 的核心抽象是 `Entity`、`Dao`、`Database`、`Migration`；
- Hilt 的核心抽象是依赖、组件、作用域、绑定、注入入口；
- Paging 的核心抽象是 `PagingSource`、`Pager`、`PagingData`、`LoadState`、`RemoteMediator`。

核心抽象越清楚，源码阅读越不容易走散。

### 3. 它如何接入工程

一个框架的接入方式通常能反映它的设计边界。

你需要观察：

- 是否需要 Gradle 插件；
- 是否需要注解处理、KAPT 或 KSP；
- 是否需要初始化；
- 是否依赖 Android 生命周期；
- 是否需要和 ViewModel、Repository、Activity、Fragment、Compose 组合；
- 是否提供测试替换方案；
- 是否适合多模块项目。

例如 Hilt 需要 Gradle 插件和注解处理，因为它要在编译期生成依赖图；Retrofit 不需要编译期插件，因为它主要依赖运行时动态代理和适配器机制；Room 需要注解处理或 KSP，因为它要根据 DAO 和 Entity 生成数据库访问代码。

### 4. 它的边界在哪里

框架解决问题，但不会解决所有问题。学习框架必须知道它的职责边界。

例如：

- OkHttp 负责 HTTP 请求执行，但不负责把接口方法变成请求；
- Retrofit 负责接口代理和响应转换，但底层网络执行通常交给 OkHttp；
- Room 负责本地数据库访问，但不负责自动决定你的业务缓存策略；
- Hilt 负责依赖创建和注入，但不负责替你设计模块边界；
- WorkManager 负责可靠后台任务，但不适合所有实时任务。

很多错误用法不是因为框架不好，而是因为把不属于它的职责塞给了它。

## 2.1.3 推荐学习路径

学习一个框架可以按照下面的顺序进行。

```text
问题背景
  -> 最小使用
  -> 核心抽象
  -> 工程接入
  -> 调用链
  -> 扩展点
  -> 常见错误
  -> 源码主线
  -> 选型判断
```

这个顺序适合大多数 Android 框架。它的好处是从“为什么用”开始，而不是直接掉进 API 细节。

## 2.1.4 第一步：阅读 README 和官方文档

README 通常不是完整教程，但它能告诉你框架作者希望你如何理解这个项目。

阅读 README 时重点看这些内容：

- 一句话介绍：框架究竟解决什么问题；
- Quick Start：最小可运行代码是什么；
- Installation：依赖、插件和版本要求；
- Core Concepts：框架作者定义了哪些核心概念；
- Samples：作者推荐的典型用法；
- FAQ：高频误区和边界条件；
- Migration Guide：框架重大版本升级时改变了什么；
- License：是否适合项目使用。

不要只复制 Quick Start。你要从 Quick Start 里反推出框架的核心模型。

以图片加载框架为例，一段简单的图片加载代码背后通常包含：

```text
Image Request
  -> Request Manager / ImageLoader
  -> Memory Cache
  -> Disk Cache
  -> Fetcher
  -> Decoder
  -> Target
```

示例代码越简单，越要追问：它隐藏了哪些复杂工作？

## 2.1.5 第二步：运行最小 Demo

学习框架时，一定要先写一个最小 Demo。Demo 不需要漂亮，但要可观察。

最小 Demo 的目标是：

- 验证依赖和版本是否可用；
- 跑通框架的核心路径；
- 观察成功、失败、取消、重试等基本状态；
- 为后续源码调试提供入口。

例如学习 Retrofit，可以先写一个只包含一个接口的 Demo：

```kotlin
interface UserApi {
    @GET("users/{id}")
    suspend fun getUser(@Path("id") id: String): UserDto
}
```

然后观察：

- Retrofit 如何创建 `UserApi`；
- `@GET` 和 `@Path` 如何被解析；
- `suspend` 方法如何适配；
- OkHttp 在哪里真正发起请求；
- JSON 在哪里被转换成 `UserDto`。

最小 Demo 不只是为了“跑起来”，而是为了找到后续分析的入口点。

## 2.1.6 第三步：识别核心入口类

每个框架都有几个非常关键的入口类。找到入口类，就找到了源码阅读的门。

| 框架 | 常见入口 |
| --- | --- |
| Coil | `ImageLoader`、`ImageRequest`、Compose 中的图片加载 API |
| Glide | `Glide.with()`、`RequestManager`、`RequestBuilder` |
| OkHttp | `OkHttpClient`、`Request`、`Call`、`Interceptor` |
| Retrofit | `Retrofit.Builder`、`Retrofit.create()`、接口代理 |
| Room | `Room.databaseBuilder()`、`@Database`、`@Dao` |
| Hilt | `@HiltAndroidApp`、`@AndroidEntryPoint`、`@Inject`、`@Module` |
| WorkManager | `WorkManager`、`Worker`、`WorkRequest`、`WorkInfo` |
| Paging 3 | `PagingSource`、`Pager`、`PagingData` |

阅读源码时，不要从工具类开始，不要从异常类开始，也不要从平台兼容类开始。优先从用户代码第一次调用框架的位置开始。

## 2.1.7 第四步：画调用链

源码阅读最容易出现的问题是“越看越多”。解决方法是画一条主调用链。

例如 Retrofit 的核心调用链可以先简化为：

```text
Retrofit.create()
  -> Proxy.newProxyInstance()
  -> InvocationHandler.invoke()
  -> ServiceMethod
  -> RequestFactory
  -> OkHttpCall
  -> CallAdapter
  -> Converter
```

OkHttp 的请求执行链可以先简化为：

```text
OkHttpClient.newCall(request)
  -> RealCall
  -> Dispatcher
  -> Interceptor Chain
  -> Connection
  -> Network I/O
  -> Response
```

Room 的代码生成主线可以先简化为：

```text
@Entity / @Dao / @Database
  -> Annotation Processor / KSP
  -> Generated DAO Implementation
  -> SQLite Statement
  -> Entity Mapping
```

调用链不需要一开始就完整。它的作用是帮你知道“我现在读到哪一层了”。

## 2.1.8 第五步：看扩展点

一个成熟框架通常会提供扩展点。扩展点能反映框架作者认为哪些部分应该开放，哪些部分应该隐藏。

常见扩展点包括：

| 扩展点类型 | 框架例子 | 作用 |
| --- | --- | --- |
| 拦截器 | OkHttp `Interceptor` | 修改请求、记录日志、统一鉴权、重试 |
| 转换器 | Retrofit `Converter` | 将响应体转换为业务对象 |
| 适配器 | Retrofit `CallAdapter` | 将请求结果适配为 suspend、Flow 或其他类型 |
| 数据源 | Paging `PagingSource` | 定义分页数据从哪里来 |
| 迁移 | Room `Migration` | 控制数据库版本升级 |
| 绑定模块 | Hilt `@Module` / `@Binds` / `@Provides` | 声明依赖如何创建 |
| 图片组件 | Coil Fetcher / Decoder | 支持新的图片来源或解码方式 |

学习扩展点时要问：

- 扩展点在哪里被调用；
- 扩展点运行在哪个线程；
- 扩展点是否能访问上下文；
- 扩展点异常如何处理；
- 扩展点是否影响性能；
- 扩展点是否容易测试。

扩展点是从“使用框架”走向“驾驭框架”的关键。

## 2.1.9 第六步：阅读测试和示例工程

很多人忽略测试，但开源项目的测试往往很有价值。

测试可以告诉你：

- 框架作者认为哪些行为必须稳定；
- 边界条件如何处理；
- 异常场景是否有明确语义；
- 某个 API 的真实使用方式；
- 框架如何支持测试替换。

示例工程则更接近真实使用场景。阅读 Sample 时要重点看：

- 项目如何分层；
- 框架对象在哪里创建；
- 是否有统一封装；
- 错误状态如何传递到 UI；
- 是否结合生命周期；
- 测试如何写。

如果 README 告诉你“怎么开始”，测试和 Sample 往往告诉你“怎么正确地长期使用”。

## 2.1.10 第七步：看 Issue、Release 和迁移文档

一个框架是否值得投入学习，不只看 API，还要看维护状态。

你可以观察：

- 最近是否有版本发布；
- Issue 是否长期无人响应；
- 是否有明确的版本升级说明；
- 是否频繁出现破坏性改动；
- 是否支持当前 Android、Kotlin、Gradle、Compose 生态；
- 社区是否有足够多的真实使用反馈；
- 安全或兼容性问题是否能及时修复。

迁移文档尤其重要。它能告诉你框架作者认为哪些设计已经过时，哪些方向是未来主线。

例如一个框架从 Java API 迁移到 Kotlin API，从回调迁移到协程，从反射迁移到代码生成，这些变化都体现了框架设计方向。

## 2.1.11 不同阶段应该读多少源码

源码不是越早读越好，也不是越多越好。可以按阶段安排。

| 阶段 | 阅读重点 | 不建议做的事 |
| --- | --- | --- |
| 初学阶段 | 入口类、核心抽象、主调用链 | 无差别通读所有模块 |
| 进阶阶段 | 缓存、线程、生命周期、扩展点 | 只看实现，不结合实际问题 |
| 工程阶段 | 错误处理、性能、测试、迁移、二次封装 | 为了“高级感”过度封装 |
| 源码专题阶段 | 设计模式、代码生成、并发模型、插件机制 | 把源码细节当成唯一目标 |

如果你是为了业务开发，优先读主线；如果你是为了框架设计能力，再深入实现细节。

## 2.1.12 推荐的框架学习笔记模板

为了让后续学习更系统，可以为每个框架整理一份固定模板。

```text
框架名称：

1. 它解决什么问题：

2. 不用它会怎么做：

3. 核心抽象：

4. 最小使用示例：

5. 工程接入方式：

6. 核心调用链：

7. 关键扩展点：

8. 常见错误：

9. 与同类框架比较：

10. 适合使用的场景：

11. 不适合使用的场景：

12. 源码入口：

13. 测试方式：

14. 迁移和维护风险：
```

这个模板会在后续章节反复使用。它能帮助你把不同框架放进同一套分析维度中，而不是每次都从零开始。

## 2.1.13 常见学习误区

### 误区一：把 API 记忆当成框架学习

API 会变，问题域和设计思想更稳定。你可以忘记某个方法名，但要知道框架为什么需要缓存、线程调度、生命周期绑定、代码生成或动态代理。

### 误区二：一开始就读全部源码

大多数框架都有大量平台适配、兼容处理和边界代码。一开始全读，往往会掩盖主线。先抓调用链，再补细节。

### 误区三：只看成功路径

真实项目中更重要的是失败路径：

- 网络失败如何重试；
- 请求取消是否及时；
- 数据库迁移失败会怎样；
- 协程异常是否会取消父任务；
- 图片加载失败是否会错位；
- WorkManager 任务失败是否会重试。

### 误区四：忽略生命周期

Android 框架与普通 Java/Kotlin 库最大的不同，是它经常和 Activity、Fragment、ViewModel、Compose、Service、进程状态绑定。忽略生命周期，很多框架都会被用错。

### 误区五：脱离工程选型

学习框架最终要服务项目。一个框架再优秀，也要考虑团队熟悉度、迁移成本、稳定性、测试方式和长期维护。

## 2.1.14 本节小结

学习一个 Android 开源框架，可以遵循这样的路线：

```text
先问问题
  -> 跑最小 Demo
  -> 抓核心抽象
  -> 找入口类
  -> 画调用链
  -> 看扩展点
  -> 读测试和 Sample
  -> 看 Issue 与迁移文档
  -> 回到工程选型
```

下一节我们会继续讨论：如何评价和选择一个开源框架。也就是说，当你已经知道一个框架“是什么”和“怎么工作”之后，如何判断它是否适合当前项目。

## 2.1.15 思考题

1. 任选一个你熟悉的 Android 框架，用“四个问题”描述它：解决什么问题、核心抽象是什么、如何接入工程、边界在哪里。
2. 为什么源码阅读应该从入口类和调用链开始，而不是从工具类或底层实现开始？
3. 选择一个框架，尝试画出它的最小调用链。
4. 阅读一个框架的 README 和 Sample，看看两者提供的信息有什么不同。
5. 找一个你项目中使用过的框架，查看它最近的 Release 和 Issue，判断它的维护状态。
