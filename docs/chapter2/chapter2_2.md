# 第 2.2 节：框架评价与选型维度

上一节讨论了如何学习一个开源框架。本节进一步讨论一个更贴近真实工程的问题：当多个框架都能解决同一类问题时，应该如何评价和选择？

框架选型不是“哪个最流行就用哪个”，也不是“哪个 API 最简单就用哪个”。真正的选型需要同时考虑业务场景、团队能力、项目阶段、维护周期、性能要求、测试方式和迁移成本。

一个成熟的 Android 工程团队，应该能够回答：

- 为什么选择这个框架；
- 为什么不选择另一个框架；
- 这个框架的风险是什么；
- 如果未来要替换，成本在哪里；
- 是否需要对框架做统一封装；
- 如何验证选型是否适合项目。

## 2.2.1 本节学习目标

学习完本节后，你应该能够：

- 建立一套通用的框架评价维度；
- 区分功能匹配、工程匹配和团队匹配；
- 使用矩阵方式比较同类框架；
- 理解选型中的性能、包体积、生命周期、测试、维护和迁移风险；
- 为项目制定更稳妥的框架引入和治理策略。

## 2.2.2 选型先看场景

框架没有脱离场景的“最好”。同一个框架，在不同项目中可能是好选择，也可能是负担。

选型前先明确项目场景：

| 问题 | 说明 |
| --- | --- |
| 项目类型是什么 | 新项目、老项目、Demo、课程项目、企业 App、SDK、车载应用 |
| 技术栈是什么 | Java、Kotlin、XML View、Compose、多模块、KMP |
| 团队熟悉什么 | 团队已有经验会直接影响学习成本和维护成本 |
| 业务复杂度如何 | 简单工具、内容列表、电商、IM、音视频、离线同步、后台任务 |
| 性能要求如何 | 启动速度、滚动流畅度、内存占用、弱网、包体积 |
| 生命周期多长 | 短期实验项目和长期维护项目的选型标准不同 |
| 测试要求如何 | 是否需要单元测试、集成测试、UI 测试、Mock 和依赖替换 |
| 是否需要迁移 | 老项目引入新框架时，迁移成本往往比框架能力更关键 |

例如一个课程 Demo 可以优先选择上手简单、概念清晰的方案；一个企业长期维护项目则要更看重稳定性、测试能力、迁移方案和团队治理。

## 2.2.3 通用评价维度

可以从以下维度评价一个开源框架。

| 维度 | 关注问题 |
| --- | --- |
| 问题匹配度 | 它是否真正解决当前项目的问题，还是为了追新而引入 |
| API 设计 | 是否清晰、类型安全、易读、易封装、符合 Kotlin 和 Android 习惯 |
| Kotlin 支持 | 是否支持 suspend、Flow、扩展函数、空安全、KSP 等现代 Kotlin 能力 |
| Compose 支持 | 是否适合 Compose 状态模型和声明式 UI |
| 生命周期感知 | 是否能正确处理 Activity、Fragment、ViewModel、Compose 生命周期 |
| 线程模型 | 是否明确运行在哪些线程，是否支持取消、超时和异常处理 |
| 性能表现 | 对启动、内存、CPU、滚动、网络、数据库和构建速度的影响 |
| 包体积 | 引入后 APK/AAB 体积是否可接受，是否依赖过多传递库 |
| 稳定性 | 是否经过大量项目验证，版本是否稳定，是否有破坏性改动 |
| 维护活跃度 | 是否持续更新，Issue 和 PR 是否有人处理 |
| 文档质量 | README、官方文档、Sample、迁移指南是否完整 |
| 可测试性 | 是否容易 Mock、替换依赖、写单元测试和集成测试 |
| 可观测性 | 是否容易记录日志、追踪错误、监控性能和定位问题 |
| 扩展性 | 是否提供清晰扩展点，是否允许自定义缓存、转换器、适配器或调度策略 |
| 迁移成本 | 从旧框架迁移过来是否可分阶段进行，是否会大面积改业务代码 |
| 团队成本 | 团队是否能理解和维护，是否需要额外培训 |
| 生态兼容 | 是否兼容当前 Android Gradle Plugin、Kotlin、Compose、Jetpack 版本 |
| 许可证 | 是否满足公司或项目的开源合规要求 |

这些维度不一定每次都全部展开。对于核心基础框架，例如网络、数据库、依赖注入和异步框架，应该评估得更严格；对于小工具库，可以简化。

## 2.2.4 功能匹配、工程匹配、团队匹配

选型可以分成三层。

### 功能匹配

功能匹配关注框架“能不能做”。

例如图片加载框架是否支持：

- 网络图片；
- 本地图片；
- 占位图和错误图；
- 内存缓存和磁盘缓存；
- GIF、SVG、视频帧；
- RecyclerView 或 LazyColumn 列表复用；
- Compose；
- 请求取消和预加载。

功能匹配是基础，但不是全部。很多框架功能都足够，真正的差异在工程匹配和团队匹配。

### 工程匹配

工程匹配关注框架“能不能长期放进项目里”。

你需要考虑：

- 是否适合项目架构；
- 是否容易封装；
- 是否容易测试；
- 是否会污染业务代码；
- 是否影响构建速度；
- 是否支持多模块；
- 是否方便升级；
- 是否能与日志、监控、异常处理体系结合。

例如 Hilt 功能强大，但它会深度进入项目的构建和依赖图；Room 能提升数据库安全性，但迁移管理需要严格规范；Paging 3 能解决分页问题，但需要团队理解 `PagingData`、`LoadState` 和数据源边界。

### 团队匹配

团队匹配关注“人能不能维护”。

同一个技术方案，对不同团队的成本不同：

- 团队是否熟悉 Kotlin 协程；
- 是否有人理解 Dagger/Hilt 的依赖图；
- 是否能处理 Room Migration；
- 是否能写 Flow 和 Paging 测试；
- 是否愿意维护统一封装；
- 是否有时间迁移老代码。

技术选型不是单纯的技术问题，也是一种团队协作问题。

## 2.2.5 使用选型矩阵

当多个框架都能解决问题时，可以用矩阵进行比较。

例如图片加载框架可以这样比较：

| 维度 | Coil | Glide | Fresco |
| --- | --- | --- | --- |
| Kotlin 友好性 | 高 | 中 | 中 |
| Compose 支持 | 高 | 可支持 | 较弱 |
| 传统 View 项目成熟度 | 高 | 很高 | 高 |
| 功能完整度 | 高 | 很高 | 高 |
| 学习成本 | 较低 | 中 | 较高 |
| 包体积与依赖 | 较轻 | 中 | 较重 |
| 存量项目使用率 | 中 | 很高 | 中 |
| 推荐场景 | Kotlin / Compose 新项目 | 传统 View 或复杂图片场景 | 特殊大图和图片管线场景 |

这个表不是为了给出绝对答案，而是让团队知道为什么做某个选择。

再例如依赖注入框架：

| 维度 | 手动 DI | Hilt | Dagger 2 | Koin |
| --- | --- | --- | --- | --- |
| 学习成本 | 低 | 中 | 高 | 低 |
| 编译期安全 | 依赖手写 | 高 | 高 | 较低 |
| Android 集成 | 手动处理 | 很好 | 需要较多配置 | 较好 |
| 运行时开销 | 低 | 低 | 低 | 相对更高 |
| 大型项目适配 | 中 | 高 | 很高 | 中 |
| 测试替换 | 手动灵活 | 官方支持较好 | 可支持 | 较灵活 |
| 推荐场景 | 小项目、教学、理解原理 | 标准 Android 项目 | 大型复杂依赖图 | Kotlin/KMP 或中小项目 |

矩阵的价值不在于“算分”，而在于把隐性判断显性化。

## 2.2.6 Must / Should / Could

为了避免选型陷入无休止争论，可以把需求分成三类。

| 分类 | 含义 | 例子 |
| --- | --- | --- |
| Must | 必须满足，不满足就不能选 | 支持当前 Kotlin 版本；支持 Compose；可测试；许可证合规 |
| Should | 应该满足，影响长期维护 | 文档完善；社区活跃；迁移指南清晰；扩展点稳定 |
| Could | 有更好，没有也能接受 | API 更优雅；示例更多；生态插件丰富 |

例如一个 Compose 新项目选择图片框架：

- Must：支持 Compose、协程、基础缓存、请求取消；
- Should：文档清晰、包体积合理、与 OkHttp 配合良好；
- Could：支持 SVG、视频帧、跨平台扩展。

这样讨论会更具体，不容易变成“我喜欢 A，你喜欢 B”。

## 2.2.7 做小规模验证

重要框架不要只看文档决定，最好做一个小规模验证。

验证内容可以包括：

- 接入是否顺利；
- 是否能跑通核心业务场景；
- 是否容易封装；
- 错误状态是否好处理；
- 是否能写单元测试；
- 是否影响构建速度；
- 是否有明显性能问题；
- 是否能和现有架构共存。

例如评估 Paging 3 时，不要只看列表是否能展示，还要验证：

- 首次加载、追加加载、刷新、重试是否符合预期；
- 网络错误如何展示；
- 数据库缓存和网络同步是否一致；
- `RemoteMediator` 是否适合你的分页接口；
- UI 层是否能清晰处理 `LoadState`；
- 测试是否可写。

小规模验证的目的不是写完整业务，而是在成本可控的情况下提前暴露风险。

## 2.2.8 关注迁移成本

很多项目不是从零开始，而是在已有代码中引入新框架。此时迁移成本往往比框架能力更重要。

迁移成本包括：

- API 替换成本；
- 数据模型修改成本；
- 线程模型变化；
- 错误处理方式变化；
- 测试重写成本；
- 团队学习成本；
- 灰度发布和回滚成本；
- 老代码与新代码共存成本。

例如从 RxJava 迁移到 Coroutines + Flow，不只是把 `Observable` 换成 `Flow`：

- 单次请求可能从 `Single` 变成 `suspend`；
- 连续数据可能从 `Observable` 变成 `Flow`；
- 线程切换从 `Scheduler` 变成 `Dispatcher`；
- 取消机制从 `Disposable` 变成 `Job`；
- 错误处理和生命周期收集方式都会变化；
- 测试工具也可能从 RxJava TestObserver 变成 Turbine 和 Coroutine Test。

迁移应该尽量分阶段进行，而不是一次性重写全部代码。

## 2.2.9 是否需要二次封装

企业项目经常会对框架做一层封装。封装的目的不是把所有 API 包起来，而是统一项目约定。

适合封装的内容：

- 网络错误模型；
- Token 和鉴权逻辑；
- 日志和监控；
- 数据库迁移规范；
- WorkManager 任务命名和重试策略；
- 图片加载占位图和错误图；
- 依赖注入模块约定；
- 测试替换入口。

不适合封装的内容：

- 为了“看起来统一”复制框架所有 API；
- 屏蔽框架关键能力，导致使用者无法处理特殊场景；
- 过度抽象，增加理解成本；
- 没有测试保障的复杂封装。

好的封装应该让常见场景更简单，让特殊场景仍然可扩展。

## 2.2.10 典型选型示例

### 示例一：新 Compose 项目选择图片框架

如果项目主要使用 Kotlin 和 Compose，通常可以优先考虑 Coil：

- Kotlin 友好；
- Compose 支持自然；
- 与协程生态配合好；
- API 简洁；
- 适合大多数常规图片加载场景。

如果项目是传统 View 存量项目，并且已经大量使用 Glide，则不一定需要迁移。Glide 依然成熟稳定，生态和历史项目经验丰富。

### 示例二：网络层选择 Retrofit + OkHttp

大多数 Android 项目会使用 Retrofit + OkHttp 的组合：

- OkHttp 负责底层 HTTP 能力；
- Retrofit 负责接口声明、请求构建和响应转换；
- Converter 负责 JSON 转换；
- CallAdapter 或 suspend 支持异步适配；
- Interceptor 支持日志、鉴权、重试和监控。

这套组合的优势是职责边界清晰，生态成熟，团队接受度高。

### 示例三：依赖注入选择 Hilt 还是手动 DI

如果项目很小，手动 DI 足够清晰，不一定要立刻引入 Hilt。

如果项目中已经有多个 Repository、UseCase、DataSource、ViewModel、Worker，并且需要测试替换和多模块管理，Hilt 会更适合。

判断标准不是“DI 框架是否先进”，而是项目复杂度是否已经需要依赖图管理。

### 示例四：是否从 RxJava 迁移到 Coroutines + Flow

新项目可以优先使用 Coroutines + Flow。它更贴近现代 Kotlin 和 Jetpack 生态。

但存量项目迁移要谨慎：

- 如果 RxJava 使用范围很小，可以逐步替换；
- 如果网络、数据库、事件流、业务编排都深度依赖 RxJava，应先建立边界；
- 可以先在新模块中使用协程，再通过适配层和老代码共存；
- 不建议为了“统一技术栈”一次性重写核心业务。

## 2.2.11 框架治理

当项目变大后，框架选型不再是一次性动作，而是持续治理。

框架治理包括：

- 建立统一依赖版本管理；
- 明确哪些框架是推荐方案，哪些是历史方案；
- 禁止同类框架无序并存；
- 为核心框架提供使用规范；
- 记录重要选型决策；
- 定期升级依赖并验证兼容性；
- 为迁移提供阶段计划；
- 对网络、存储、图片、日志、测试等基础能力建立统一入口。

例如同一个项目里同时出现 Gson、Moshi、Kotlinx Serialization，或者同时出现 Volley、OkHttp、多个自研网络库，就会增加维护成本。治理的目的不是限制开发，而是让长期维护更稳定。

## 2.2.12 选型检查清单

引入一个重要框架前，可以检查：

```text
1. 它解决的问题是否真实存在？
2. 是否已有框架或平台能力可以解决？
3. 是否适合当前项目技术栈？
4. 是否支持 Kotlin、Compose、Jetpack 生命周期？
5. 是否容易测试？
6. 是否容易监控和排查问题？
7. 是否会显著增加包体积或构建成本？
8. 是否有活跃维护和清晰文档？
9. 是否符合许可证要求？
10. 是否有迁移和回滚方案？
11. 是否需要二次封装？
12. 团队是否具备维护能力？
```

如果一个框架无法回答这些问题，就不适合贸然引入核心工程。

## 2.2.13 本节小结

框架选型不是追求“最流行”，而是寻找“最适合当前项目”的方案。

本节建立了几个关键判断：

- 选型必须从场景出发；
- 评价框架要同时看功能、工程和团队；
- 选型矩阵可以把隐性判断显性化；
- 重要框架需要小规模验证；
- 存量项目要重点关注迁移成本；
- 二次封装应该服务项目约定，而不是制造额外复杂度；
- 大型项目需要持续的框架治理。

从下一章开始，我们会进入第一个具体问题域：图片加载框架。你会看到本章的方法如何真正用于分析 Coil、Glide、Picasso 和 Fresco。

## 2.2.14 思考题

1. 你的项目中是否存在同类框架并存的情况？这种并存是否合理？
2. 选择图片加载框架时，你认为哪些维度是 Must，哪些是 Should？
3. 如果一个框架 API 很简单，但维护不活跃，你会如何判断是否引入？
4. 为什么企业项目经常需要对网络层做统一封装，而不直接在业务代码里到处使用 Retrofit？
5. 如果要从 RxJava 迁移到 Coroutines + Flow，你会如何设计分阶段迁移方案？
