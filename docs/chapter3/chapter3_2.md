# 第 3.2 节：缓存系统与缓存 Key

“框架有内存缓存和磁盘缓存”只是入门结论。真正需要掌握的是：每层缓存保存什么、Key 如何生成、查询顺序为何不同，以及图片内容变化时怎样让旧结果自然失效。

> 开局预测：一张图片首次加载后，依次发生“再次进入页面、杀死进程、修改圆角、修改版本”时，分别会命中哪一层？可以先到 [缓存管线 Demo](./chapter3_7.md#_3-7-3-挑战一-让同一-url-连续走四条不同路径) 试一次。

## 3.2.1 先区分四种容易混淆的缓存

| 缓存 | 常见内容 | 读取成本 | 典型用途 |
| --- | --- | --- | --- |
| 活跃资源 | 正在被某个 Target 使用的已解码资源 | 最低 | 多个目标显示同一结果 |
| 内存缓存 | 最近使用的已解码或已变换图片 | 很低 | 返回列表、重复显示 |
| 图片磁盘缓存 | 原始字节或变换后的资源文件 | 中 | 避免重新下载或重新解码 |
| HTTP 缓存 | 遵循响应头保存的 HTTP 响应 | 中 | 条件请求、减少传输 |

图片框架的磁盘缓存和 OkHttp 的 HTTP 缓存不是同一层。前者按图片请求与转换结果组织，后者按 HTTP 语义处理 `Cache-Control`、`ETag` 等响应信息。项目可能同时使用两者，也可能只启用其中一种。

## 3.2.2 Glide 的四级查询说明了什么

Glide 官方缓存文档把默认查询顺序概括为：

```text
Active Resources
  -> Memory Cache
  -> Resource Disk Cache
  -> Data Disk Cache
  -> Original Source
```

各层回答的问题不同：

- Active Resources：这个结果是否正在被别的 Target 使用；
- Memory Cache：最近是否已经得到过完全匹配的资源；
- Resource Disk Cache：是否已有解码、缩放或变换后的资源；
- Data Disk Cache：是否保存了原始数据，可以本地重新解码；
- Original Source：都没有命中才访问文件、Uri 或网络。

为什么同时需要 Resource 与 Data 两种磁盘缓存？因为“网络昂贵”和“解码昂贵”是两类成本。保存原始数据可以生成多个尺寸；保存变换结果则能跳过解码和变换。不同来源、尺寸数量和磁盘预算会影响策略选择。

## 3.2.3 一个缓存 Key 到底由什么组成

可以把完整 Key 抽象为：

```text
CacheKey = hash(
    model identity,
    content version,
    target width and height,
    decoded resource type,
    transformations and parameters,
    decode options
)
```

不是每一层都使用完全相同的字段。例如原始数据缓存通常不需要关心圆角半径，因为原始字节没有被圆角处理；变换后的资源缓存则必须区分尺寸和变换。

### 为什么 URL 相同不代表结果相同

同一地址可能产生：

```text
72 x 72 的 centerCrop 商品封面
300 x 300 的圆角商品卡片
1080 x 1920 的 fitCenter 详情图
Bitmap 输出
GIF / Drawable 输出
```

如果这些结果共用一个资源 Key，框架就可能返回尺寸或变换不匹配的对象。因此尺寸、变换、输出类型和相关选项必须参与对应缓存层的身份计算。

## 3.2.4 Key 的两个质量要求

### 稳定

同一份内容、同一组选项必须得到相同 Key。自定义模型参与 Key 时，`equals()`、`hashCode()` 或 Keyer 不能依赖每次都会变化的对象地址和随机值。

### 可失效

内容变化后必须得到新 Key。商品图片 URL 不变时，可以混入业务版本：

```kotlin
val cacheVersion = "${product.id}:${product.imageVersion}"
```

Coil 可以显式设置缓存 Key；Glide 可以使用 `signature(ObjectKey(...))` 把版本信息混入 Key。

```kotlin
// Coil
ImageRequest.Builder(context)
    .data(product.coverUrl)
    .memoryCacheKey("cover:$cacheVersion")
    .diskCacheKey("cover:$cacheVersion")
    .build()

// Glide
Glide.with(fragment)
    .load(product.coverUrl)
    .signature(ObjectKey(cacheVersion))
    .into(imageView)
```

最优方案通常是 CDN 在内容变化时生成新 URL；做不到时再使用更新时间、内容哈希或服务端版本号。不要用“当前时间”作为版本，它会让每次请求都变成新资源，等于主动摧毁缓存命中率。

## 3.2.5 为什么不推荐按 URL 搜索并删除磁盘文件

磁盘文件名通常是完整 Key 的哈希。一个 URL 可能对应多个尺寸、变换和资源类型，因此很难可靠找到它衍生出的所有文件。

更稳妥的失效方式是版本化：

```text
旧 Key -> 旧文件仍在，但不再被请求命中 -> LRU 最终淘汰
新 Key -> 产生新结果 -> 后续稳定命中
```

这把“删除所有旧变体”的复杂操作，转换成“让新请求拥有新身份”的简单操作。

## 3.2.6 LRU 不是一句“最近最少使用”

假设内存缓存上限为 10 MB，当前有：

```text
A: 4 MB，刚刚访问
B: 3 MB，很久未访问
C: 2 MB，中等时间访问
```

现在放入 D（4 MB），总量将到 13 MB。LRU 会从最久未使用的 B 开始淘汰；剩余仍超限时继续淘汰下一个最久未使用项，直到低于上限。

LRU 优化的是时间局部性：刚使用过的图片更可能再次使用。它不理解“首页图片比二级页更重要”这类业务语义，因此不要把图片缓存当作永久业务存储。

## 3.2.7 内存缓存命中仍可能带来 UI 状态变化

“命中内存”不等于业务层一定看不到 Loading 状态。Compose 的状态建立、首帧绘制和 painter 状态通知可能发生在不同时间点；某些 API 在首次 composition 中即使能够绘制内存结果，公开 state 仍可能先经历初始状态。

因此 UI 不应假设：

```text
只要缓存命中，状态就一定同步从 Empty 直接变 Success
```

设计占位图和过渡效果时，应以具体 API 的状态约定为准，并用实际首帧实验验证是否闪烁。

## 3.2.8 缓存策略不是越多越好

按业务类型做判断：

| 图片类型 | 建议思路 | 原因 |
| --- | --- | --- |
| 商品图、文章图 | 内存 + 磁盘，内容版本化 | 重复访问多，内容相对稳定 |
| 头像 | 缓存并混入头像版本 | URL 可能长期不变但内容会更新 |
| 验证码 | 禁用或严格隔离缓存 | 每次内容必须不同 |
| 带隐私的临时图片 | 谨慎使用磁盘缓存 | 需要符合安全与清理要求 |
| 本地原图编辑预览 | 关注文件修改时间 | 路径不变但文件会被覆盖 |

对于带鉴权参数的 URL，还要注意 token 是否会进入 Key、日志和磁盘文件映射。可以把“请求地址”和“缓存身份”解耦：网络仍使用签名 URL，缓存使用稳定且不包含敏感信息的业务 Key。

## 3.2.9 缓存侦探实验

### 准备观测字段

每次请求记录：

```text
requestId
model
resolvedSize
cacheKey / version
dataSource
startTime / endTime
result
```

### 实验步骤

1. 清空应用数据，首次进入商品列表；
2. 记录网络请求数、耗时和结果来源；
3. 返回上一页后再次进入，观察内存命中；
4. 杀死进程后重进，观察磁盘命中；
5. 修改 `imageVersion` 但保持 URL 不变；
6. 修改目标尺寸或圆角参数；
7. 比较每次请求的 Key、数据来源和耗时。

### 预期推理

```text
首次加载：网络 / 原始来源
同进程返回：优先内存
重启应用：内存消失，可能命中磁盘
版本变化：旧 Key 不命中
尺寸或变换变化：原始数据可能命中，变换结果不一定命中
```

如果观测结果与推理不同，不要先怪框架。依次检查目标尺寸是否稳定、Key 是否真的相同、变换参数是否一致、缓存策略是否关闭、响应是否允许保存。

## 3.2.10 本节检查点

1. 图片磁盘缓存和 HTTP 缓存分别解决什么问题？
2. 为什么原始数据缓存通常不需要包含圆角参数，而资源缓存需要？
3. 为什么“清空全部缓存”通常不是图片更新的正确方案？
4. 自定义模型能成功加载，却总不命中内存缓存，最先应该检查什么？
5. 什么时候应该让请求地址与缓存身份分离？

<details>
<summary>检查答案</summary>

1. 图片磁盘缓存复用编码数据或变换资源，HTTP 缓存按 HTTP 响应语义复用网络响应；
2. 圆角不改变原始字节，却会改变最终资源像素；
3. 使用新版本生成新 Key 更稳定，旧条目可由 LRU 自然淘汰；
4. 检查自定义模型的 Keyer、`equals()`、`hashCode()` 与请求尺寸是否稳定；
5. 签名 URL 含临时 token 或敏感参数，而内容本身有稳定业务身份时应分离。

</details>
