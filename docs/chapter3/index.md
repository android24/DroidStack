# 第 3 章：图片加载框架

第一章建立了 Android 开源框架全景，第二章给出了一套学习和评价框架的方法。本章把这套方法第一次落到真实框架上：不把 Coil、Glide 当成几个图片 API，而是把它们当成由请求调度、缓存、解码、生命周期和资源管理共同组成的系统。

先看五个常见故障：

```text
商品列表快速滑动，图片串到了另一个商品上
详情页只显示一张图，内存却突然增加几十 MB
服务端替换了图片，客户端仍然显示旧内容
Fragment 已经销毁，旧请求仍尝试更新 ImageView
同一张图片被多个位置请求，却重复下载、重复解码
```

如果只能回答“加 placeholder”“调用 clear”“关闭缓存”，仍然只是会用 API。本章要继续追问：

- 旧请求为什么能覆盖新请求，框架把当前请求保存在哪里；
- URL 相同为什么不一定命中缓存，一个完整缓存 Key 包含什么；
- 200 KB 的 JPEG 为什么可能变成 45 MB 的 Bitmap；
- `Glide.with(fragment)` 为什么比传入一个 `Context` 多做了很多事；
- `Fetcher`、`Decoder`、`Target` 为什么要拆开，它们在调用链上如何协作；
- Builder、Strategy、Factory、Chain、Observer 如何真实地参与一次请求，而不是只出现在模式表格里。

## 本章学习目标

完成本章后，你应该能够：

- 画出 Coil 与 Glide 从入口 API 到结果显示的关键调用链；
- 区分内存缓存、资源磁盘缓存、原始数据缓存和网络 HTTP 缓存；
- 根据目标尺寸估算 Bitmap 内存，并解释采样、变换和复用的关系；
- 用时间线解释 RecyclerView 图片错位，并说明取消为何能够阻止错误回写；
- 把设计模式定位到真实对象、真实调用点和真实变化方向；
- 通过日志、断点和可复现实验验证自己的判断；
- 根据 UI 技术栈、业务场景和维护成本选择 Coil、Glide 或其他方案。

## 贯穿案例

所有专题都围绕同一个商品图片场景展开：

```kotlin
data class Product(
    val id: String,
    val name: String,
    val coverUrl: String,
    val detailImageUrl: String,
    val imageVersion: Long
)
```

列表页展示固定尺寸封面，详情页展示高清图。用户会快速滑动、刷新、进入详情再返回；图片地址可能不变但内容会更新；网络可能成功、超时或离线。这个案例会迫使我们同时处理缓存、尺寸、并发、生命周期和错误状态。

## 深度标准

本章不以“出现过多少名词”衡量深度，而以能否完成以下推理衡量：

| 层次 | 读者能够回答 |
| --- | --- |
| API 层 | 怎样发起请求，怎样配置占位图和错误图 |
| 角色层 | Request、Loader、Fetcher、Decoder、Cache、Target 分别负责什么 |
| 流程层 | 这些对象以什么顺序协作，成功、失败、取消分别走哪条路径 |
| 机制层 | 缓存 Key、尺寸解析、竞态消除、资源复用为什么有效 |
| 源码层 | 从哪个入口下断点，下一跳依据什么条件选择 |
| 工程层 | 如何封装、观测、测试和选型，出现故障时先查什么 |

只有从 API 层走到工程层，才算真正掌握一个图片加载框架。
