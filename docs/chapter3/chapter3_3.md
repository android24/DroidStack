# 第 3.3 节：解码、尺寸与 Bitmap 内存

图片文件很小，不代表显示时占用内存也小。要理解图片 OOM，必须把“压缩文件大小”和“解码后的像素内存”彻底分开。

## 3.3.1 200 KB 的 JPEG 为什么可能占几十 MB

JPEG、PNG、WebP 是编码后的文件；ImageView 或 Compose 最终绘制的是解码后的像素。常见 Bitmap 内存近似为：

```text
memory = decodedWidth x decodedHeight x bytesPerPixel
```

常见像素格式：

| Bitmap Config | 每像素字节数 | 特点 |
| --- | ---: | --- |
| `ARGB_8888` | 4 | 颜色质量高，常见默认选择 |
| `RGB_565` | 2 | 省内存，但颜色精度降低且不支持透明度 |
| `ALPHA_8` | 1 | 只保存透明度 |
| `RGBA_F16` | 8 | 高精度 / 广色域，内存更高 |

一张 `4000 x 3000` 的 ARGB_8888 图片：

```text
4000 x 3000 x 4 = 48,000,000 bytes ~= 45.8 MiB
```

磁盘上即使只有 2 MB，完整解码后仍接近 46 MiB。列表同时持有多张、发生变换产生临时 Bitmap、缓存保留结果时，峰值还会继续增加。

## 3.3.2 OOM 看的是峰值，不是单张结果

一次圆角或裁剪可能短暂同时存在：

```text
压缩数据缓冲区
+ 解码中的像素缓冲区
+ 原始 Bitmap
+ 变换后的 Bitmap
+ UI 正在显示的上一张 Bitmap
+ 内存缓存和 Bitmap Pool 中的其他资源
```

所以“最后显示的 Bitmap 只有 2 MB”不能证明加载过程只需要 2 MB。定位 OOM 要看请求并发量、临时对象和峰值时刻。

## 3.3.3 目标尺寸为什么必须尽早确定

列表封面只有 `144 x 144 px`，没有必要先解码 `4000 x 3000` 再缩小。框架通常先解析目标尺寸，再选择采样或解码尺寸：

```text
读取图片头，不分配完整像素
  -> 得到原图宽高与格式
  -> 得到 Target 目标宽高
  -> 结合 Scale / ContentScale 计算解码尺寸
  -> 选择采样比例
  -> 分配 Bitmap 并解码
  -> 必要时执行精确缩放和变换
```

目标尺寸越晚确定，框架越可能保守地加载更大资源。

### Compose 如何提供尺寸

```kotlin
AsyncImage(
    model = product.coverUrl,
    contentDescription = product.name,
    contentScale = ContentScale.Crop,
    modifier = Modifier.size(72.dp)
)
```

`AsyncImage` 可以结合布局约束与 `ContentScale` 推导请求尺寸。固定 `size` 或稳定 `aspectRatio` 不只是布局要求，也是在给解码器提供边界。

如果直接使用 `rememberAsyncImagePainter`，需要留意所选 API 是否自动获得布局约束；无法获得时应显式配置 `SizeResolver`，否则可能按原始尺寸加载。

### Glide 如何获得尺寸

Glide 的 `ViewTarget` 会从布局参数、View 实际尺寸或布局监听中解析目标尺寸。尺寸始终固定时，也可以显式指定：

```kotlin
Glide.with(fragment)
    .load(product.coverUrl)
    .override(144, 144)
    .centerCrop()
    .into(imageView)
```

`wrap_content` 无法提供可靠边界时，框架可能退回更大的估计尺寸。列表项尺寸应尽量稳定，既降低内存，也提高缓存和 Bitmap Pool 的复用概率。

## 3.3.4 采样比例怎样推导

假设原图 `4000 x 3000`，目标容器 `400 x 300`。最直观的比例是：

```text
widthRatio  = 4000 / 400 = 10
heightRatio = 3000 / 300 = 10
```

解码器会依据平台 API、格式和缩放策略选择接近目标的采样尺寸。教学上可以用 `inSampleSize` 的思路理解：若按 8 倍采样，得到约 `500 x 375`，再做精确缩放；如果完全按原图解码，则像素数量是目标的 100 倍。

```text
原图内存：4000 x 3000 x 4 ~= 45.8 MiB
采样后：  500 x 375 x 4   ~= 0.72 MiB
目标图：  400 x 300 x 4   ~= 0.46 MiB
```

采样不是“降低一点质量”的小优化，而可能把单张图片的内存降低两个数量级。

## 3.3.5 `centerCrop` 与 `fitCenter` 改变了什么

假设原图比例为 4:3，目标容器为 1:1：

- `fitCenter`：完整图片进入容器，短边可能留白；
- `centerCrop`：填满容器，多余区域被裁掉。

它们会影响：

1. 解码所需的最小尺寸；
2. 最终输出像素；
3. 资源缓存 Key；
4. 是否需要生成新的 Bitmap。

因此缩放模式不只是 UI 表现，它也是加载管线的一部分。

## 3.3.6 Transformation 为什么会影响缓存与峰值内存

圆角、模糊、水印等变换读取输入像素并生成输出。一个可靠 Transformation 至少要满足：

- 相同参数产生相同结果；
- 参数参与缓存 Key；
- 不在主线程执行昂贵像素操作；
- 正确处理输入、输出资源的所有权；
- 能复用 Bitmap 时不额外分配无意义对象。

如果圆角半径没有进入 Key：

```text
请求 A：radius = 8
请求 B：radius = 24
```

二者可能错误地命中同一个变换结果。Strategy 让变换算法可替换，但 Key 协议决定替换后缓存是否仍然正确。

## 3.3.7 Hardware Bitmap 与软件 Bitmap

较新 Android 版本可以使用硬件配置 Bitmap，让像素数据更适合图形管线并减少 Java / Native 堆的某些压力。但硬件 Bitmap 有限制，例如像素不可随意修改，某些需要软件 Canvas 或直接像素访问的变换无法使用。

框架通常会根据设备版本、请求尺寸、是否需要变换和目标类型决定是否允许硬件 Bitmap。工程代码不要为了“看起来更快”全局强制某种配置，应针对真实场景测量。

## 3.3.8 Bitmap Pool 解决的不是缓存命中

Glide 的 Bitmap Pool 保存的是可复用像素内存，用于后续解码；Memory Cache 保存的是可以直接返回给调用者的完整结果。

```text
Memory Cache 命中：这张目标图片已经有了，直接显示
Bitmap Pool 命中：有一块合适的内存可用于解码，但仍要读取和解码图片
```

Pool 的价值是减少频繁分配大块像素内存带来的 GC 与抖动。尺寸越稳定，越容易找到可复用缓冲区。

## 3.3.9 为什么不要手动 `recycle()` 框架管理的 Bitmap

图片框架需要知道资源何时仍被 Target、缓存或其他请求使用。业务代码提前调用 `recycle()`，框架可能仍认为资源有效，之后绘制或复用时就会出现崩溃或画面异常。

正确做法是通过框架的 Target 清理和生命周期 API 释放所有权，由框架决定进入缓存、归还 Pool 或真正释放。

## 3.3.10 大图内存账本实验

准备一张 `4000 x 3000` 图片，设计四组请求：

| 组别 | 目标 | 预期观察 |
| --- | --- | --- |
| A | 原始尺寸 | 内存与解码耗时最高 |
| B | `1080 x 810` | 明显下降 |
| C | `400 x 300` | 适合普通内容卡片 |
| D | `144 x 144` + crop | 适合列表封面，像素最少 |

每组记录：

```text
原图宽高
请求目标宽高
实际输出宽高
Bitmap Config
估算内存
加载耗时
峰值内存
数据来源
```

不要只看 Android Studio 的总内存曲线。先用公式估算单张结果，再结合并发数量和临时对象解释峰值。理论与观测不一致时，检查设备密度缩放、输出类型、动画图片帧和缓存保留。

## 3.3.11 定位图片内存问题的顺序

```text
确认原图与目标尺寸
  -> 确认实际输出尺寸和 Bitmap Config
  -> 检查列表是否同时持有过多结果
  -> 检查是否重复请求或尺寸不稳定
  -> 检查 Transformation 临时分配
  -> 检查 GIF / 动画 / 超大图等特殊类型
  -> 最后再调整缓存与 Pool 大小
```

直接缩小内存缓存可能降低 OOM 风险，却增加重复解码和卡顿；直接扩大缓存又可能抬高峰值。先找尺寸和所有权问题，再调容量。
